# Изменения, внесённые в проект

## Зачем вообще были внесены изменения

В этом чате проект был доработан под требования лабораторной работы. Требовалось не просто добавить новый endpoint, а показать несколько разных технических механизмов:

1. сложный GET-запрос с фильтрацией по вложенной сущности через JPQL;
2. аналогичный запрос через native SQL;
3. пагинацию;
4. in-memory индекс на `HashMap<K, V>` для уже выполненных запросов;
5. инвалидиацию индекса при изменении данных;
6. документацию, по которой можно показать работу преподавателю.

В качестве логичного сценария был выбран поиск ресторанов по параметрам блюд. Это естественно для предметной области ресторана: пользователь ищет ресторан не только по городу и кухне, но и по блюдам и диапазону цен.

## Что было добавлено по бизнес-логике

### Новый сценарий поиска

Добавлен поиск ресторанов с фильтрами:

- `city`
- `cuisineType`
- `dishName`
- `minDishPrice`
- `maxDishPrice`

То есть итоговый объект поиска — `Restaurant`, а фильтрация идёт в том числе по вложенной сущности `Dish`.

Это соответствует требованию лабораторной про фильтрацию по вложенной сущности.

## Какие файлы были изменены и зачем

### [RestaurantRepository.java](d:/javalabs/app/src/main/java/com/restaurant/app/repository/RestaurantRepository.java)

Что делает:

- остаётся основным Spring Data JPA репозиторием для `Restaurant`;
- теперь содержит JPQL-запрос через `@Query`;
- расширяет кастомный интерфейс `RestaurantNativeSearchRepository`.

Зачем менялся:

- сюда был добавлен сложный JPQL-запрос для поиска по `Restaurant -> Dish`;
- native-вариант из `@Query(nativeQuery = true)` был затем убран, потому что по условию его лучше показать как отдельный SQL-файл.

Почему это логично:

- JPQL удобно держать прямо в JPA-репозитории;
- это хорошо показывает именно ORM-подход.

### [RestaurantNativeSearchRepository.java](d:/javalabs/app/src/main/java/com/restaurant/app/repository/RestaurantNativeSearchRepository.java)

Что делает:

- объявляет кастомный контракт для native-поиска ресторанов.

Зачем нужен:

- `JpaRepository` хорошо работает с JPQL и простыми derived query;
- когда native SQL выносится в отдельный `.sql` файл, нужен свой код, который этот файл читает, подставляет параметры и выполняет запрос;
- для этого и создаётся отдельный кастомный контракт.

Проще говоря:

- `RestaurantRepository` отвечает за стандартный JPA-стиль;
- `RestaurantNativeSearchRepository` добавляет к нему нестандартную реализацию native SQL из файла.

### [RestaurantNativeSearchRepositoryImpl.java](d:/javalabs/app/src/main/java/com/restaurant/app/repository/RestaurantNativeSearchRepositoryImpl.java)

Что делает:

- это реализация кастомного native-поиска;
- читает SQL из файлов в `src/main/resources/sql`;
- подставляет параметры через `NamedParameterJdbcTemplate`;
- отдельно выполняет основной запрос и `count`-запрос;
- собирает `Page<Restaurant>`;
- восстанавливает итоговый список `Restaurant` по найденным `id`.

Почему этот файл вообще понадобился:

- если native SQL хранится в отдельном файле, Spring Data не умеет автоматически взять этот файл и исполнить его вместо `@Query`;
- значит, нужен явный Java-код, который:
  `1.` загрузит SQL-текст;
  `2.` передаст параметры;
  `3.` добавит `ORDER BY`, `LIMIT`, `OFFSET`;
  `4.` посчитает общее количество строк для пагинации;
  `5.` вернёт результат в формате Spring Data `Page`.

Почему нельзя было оставить всё только в `RestaurantRepository`:

- в обычном интерфейсе репозитория можно объявить метод, но не написать для него такую нестандартную логику без отдельной реализации;
- как только нужен внешний SQL-файл и ручное исполнение, появляется необходимость в кастомной реализации.

Почему реализация выглядит именно так:

- native SQL возвращает сначала `id` ресторанов;
- потом через JPA загружаются сами сущности `Restaurant`;
- это позволяет сохранить совместимость с текущей моделью сущностей и не писать вручную полный mapper результата SQL в entity.

Это немного сложнее, чем `@Query(nativeQuery = true)`, но зато:

- native SQL действительно лежит в отдельном PostgreSQL-файле;
- код явно показывает, как этот SQL исполняется;
- на защите это проще аргументировать как отдельную реализацию native query.

## Краткая логика архитектурного решения

### Почему JPQL и native сделаны по-разному

JPQL-запрос удобно держать в `@Query`, потому что:

- он связан с entity-моделью;
- он хорошо читается на уровне JPA;
- это стандартный путь для ORM-запроса.

Native SQL вынесен в отдельный файл, потому что:

- пользователь отдельно уточнил это требование;
- преподавателю проще показать реальный SQL-файл;
- это подчёркивает разницу между JPQL и native SQL.

### Почему понадобился `RestaurantNativeSearchRepositoryImpl`

Короткий ответ:

- потому что внешний `.sql` файл нельзя просто “подключить” к методу `JpaRepository` без собственной реализации.

Если совсем просто:

- `@Query` подходит, когда SQL или JPQL написан прямо в аннотации;
- отдельный SQL-файл требует Java-код, который этот файл прочитает и выполнит;
- этот Java-код и находится в `RestaurantNativeSearchRepositoryImpl`.
