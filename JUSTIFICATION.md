# Обоснование Архитектурных Решений

Этот файл нужен как краткое объяснение того, почему в проекте выбраны именно такие сущности, связи, `CascadeType` и `FetchType`.

## 1. Общая идея модели

Проект моделирует систему бронирования столов в ресторанах.

Основная предметная область разбита на сущности:

- `Restaurant`
- `RestaurantTable`
- `Dish`
- `Customer`
- `Booking`
- `BookingStatus`

Такое разделение выбрано потому, что каждая сущность отвечает за отдельную бизнес-роль, а не смешивает несколько задач в одной таблице.

## 2. Почему нужны именно эти сущности

### `Restaurant`

Это главная сущность в системе.

Она нужна для хранения:

- названия ресторана
- города
- типа кухни

Почему выделена отдельно:

- ресторан существует сам по себе
- у ресторана есть собственные данные
- к ресторану логически привязаны столы и блюда

### `RestaurantTable`

Это не соединительная таблица, а отдельная бизнес-сущность.

Она нужна, потому что стол имеет собственные свойства:

- номер стола
- количество мест

Почему это отдельная сущность:

- стол существует как самостоятельный объект внутри ресторана
- на стол можно оформить бронирование
- стол участвует в бизнес-логике, а не только соединяет две другие таблицы

### `Dish`

Эта сущность нужна для меню ресторана.

Почему она отдельная:

- блюдо имеет собственные атрибуты
- блюдо принадлежит конкретному ресторану
- блюда используются для демонстрации связи `OneToMany` и проблемы `N+1`

### `Customer`

Нужна для хранения клиента, который делает бронирование.

Почему отдельно:

- у клиента есть собственные данные
- один клиент может делать несколько бронирований
- бронирование без клиента в данной модели не имеет смысла

### `Booking`

Это сущность бронирования.

Она связывает:

- клиента
- стол
- время бронирования
- статус бронирования

Почему это отдельная сущность:

- бронирование является самостоятельным событием в системе
- у него есть время, статус и ссылки на другие сущности
- одна и та же пара `Customer` и `RestaurantTable` может встречаться много раз в разное время

### `BookingStatus`

Это enum, а не отдельная таблица.

Почему выбран enum:

- список статусов заранее известен
- статусы ограничены фиксированным набором
- для учебного проекта enum проще, чище и логичнее, чем отдельная справочная таблица

Текущие значения:

- `CREATED`
- `CONFIRMED`
- `CANCELED`

## 3. Почему такие связи между сущностями

### `Restaurant -> RestaurantTable` это `OneToMany`

Причина:

- у одного ресторана много столов
- один стол относится только к одному ресторану

Это естественная связь предметной области.

### `Restaurant -> Dish` это `OneToMany`

Причина:

- ресторан предлагает много блюд
- конкретное блюдо в этой модели относится к одному ресторану

### `Customer -> Booking` это `OneToMany`

Причина:

- один клиент может сделать несколько бронирований
- каждое конкретное бронирование принадлежит одному клиенту

### `RestaurantTable -> Booking` это `OneToMany`

Причина:

- один стол может быть забронирован много раз в разное время
- каждое конкретное бронирование относится к одному столу

## 4. Почему используется `CascadeType`

`CascadeType` нужен там, где дочерняя сущность жизненно зависит от родительской.

### `Restaurant.tables` -> `cascade = ALL`

Почему это оправдано:

- столы не имеют смысла вне ресторана
- если создаётся ресторан вместе со столами, удобно сохранить всё одной операцией
- если ресторан удаляется, его столы тоже должны удалиться

Почему `ALL`:

- покрывает сохранение, обновление и удаление
- удобно для демонстрации связанных сущностей

### `Restaurant.dishes` -> `cascade = ALL`

Почему:

- блюда принадлежат ресторану
- при создании ресторана можно сразу добавить меню
- удаление ресторана логически должно удалить и его блюда

### `RestaurantTable.bookings` -> `cascade = ALL`

Почему:

- в текущей модели бронирование зависит от существования стола
- при удалении стола связанные бронирования не должны оставаться "висячими"

### Почему у `Customer.bookings` нет cascade

Это осознанное решение.

Причина:

- клиент и его история бронирований не всегда должны жёстко удаляться одной операцией
- с точки зрения бизнес-логики это уже более спорное место
- для учебного проекта лучше показать, что cascade применяется не везде автоматически, а только там, где это логично

То есть:

- `CascadeType` используется не "везде подряд"
- он включён только там, где дочерние сущности действительно принадлежат родителю

## 5. Почему используется `orphanRemoval = true`

`orphanRemoval` нужен там, где удалённый из коллекции объект не должен продолжать жить отдельно.

### Для `Restaurant.tables`

Если стол убрали из ресторана, он не должен остаться в базе без ресторана.

### Для `Restaurant.dishes`

Если блюдо убрали из меню ресторана, оно не должно остаться как отдельная запись без владельца.

### Для `RestaurantTable.bookings`

Если бронирование убрали из коллекции стола, в этой модели оно тоже должно удаляться как зависимая запись.

## 6. Почему используется `FetchType.LAZY`

`LAZY` выбран почти везде специально.

Причина:

- связанные коллекции не всегда нужны
- если всегда тянуть всё сразу, приложение начнёт выполнять лишние SQL-запросы или собирать слишком тяжёлые объекты
- `LAZY` даёт контроль над загрузкой данных

На защите это можно объяснить так:

- по умолчанию загружаем только то, что реально нужно в текущем сценарии
- если требуется связанный граф объектов, загружаем его явно через `JOIN FETCH` или `EntityGraph`

## 7. Почему `LAZY` особенно важен в этом проекте

Этот проект специально показывает проблему `N+1`.

Если бы связи были `EAGER`, то:

- демонстрация `N+1` была бы менее наглядной
- Hibernate мог бы тянуть слишком много данных автоматически
- поведение стало бы менее контролируемым

Поэтому `LAZY` здесь полезен не только архитектурно, но и учебно.

## 8. Почему не выбран `EAGER`

`EAGER` удобен только на первый взгляд.

Его минусы:

- данные могут грузиться даже тогда, когда они не нужны
- сложнее контролировать SQL
- выше риск лишних запросов и перегруженных объектов
- при нескольких связях `EAGER` быстро начинает вредить производительности

Поэтому в проекте выбран подход:

- базовые связи `LAZY`
- оптимизация загрузки в нужных местах через специальные запросы

## 9. Почему для решения `N+1` используются `JOIN FETCH` и `EntityGraph`

### `JOIN FETCH`

Подходит, когда нужно явно загрузить конкретную связанную коллекцию в одном запросе.

В проекте он используется для сценария:

- `Restaurant -> Dish`

Почему это хорошо:

- решение простое
- SQL легко объяснить
- хорошо подходит для демонстрации

### `EntityGraph`

Подходит, когда нужно загрузить несколько уровней связей.

В проекте он используется для сценария:

- `Restaurant -> RestaurantTable -> Booking`

Почему это хорошо:

- декларативный способ управления загрузкой
- удобен для вложенных связей
- хорошо показывает, что проблему можно решать не только через `JOIN FETCH`

## 10. Почему транзакции вынесены в отдельный демонстрационный блок

Транзакции в проекте показаны отдельно, потому что задача учебная и нужно явно продемонстрировать:

- частичное сохранение без `@Transactional`
- откат при исключении с `@Transactional`

Это вынесено в отдельные endpoint-ы, чтобы:

- поведение было наглядным
- можно было быстро проверить счётчики записей в базе
- преподавателю было легко увидеть разницу между двумя сценариями

## 11. Как это кратко объяснять преподавателю

Можно отвечать так:

1. Сущности выделены по бизнес-смыслу: ресторан, стол, блюдо, клиент, бронирование.
2. `RestaurantTable` не является соединительной таблицей, потому что у него есть собственные поля и собственная логика.
3. `CascadeType.ALL` включён там, где дочерние сущности принадлежат родителю по жизненному циклу.
4. `FetchType.LAZY` выбран для контроля загрузки и для предотвращения лишних запросов.
5. Проблема `N+1` в проекте показана специально, а затем решена через `JOIN FETCH` и `EntityGraph`.
6. Enum `BookingStatus` выбран потому, что набор статусов фиксированный и не требует отдельной таблицы.

## 12. Короткий вывод

Архитектура модели выбрана не случайно.

Она позволяет:

- логично отразить предметную область
- показать связи между сущностями
- продемонстрировать каскадное сохранение
- продемонстрировать транзакции
- продемонстрировать проблему `N+1` и её решение

То есть модель одновременно подходит и для реальной логики, и для учебной демонстрации важных возможностей JPA/Hibernate.
